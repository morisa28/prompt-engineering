# Report Writing

## 1. Purpose

用于把事实材料组织成面向特定读者、目的明确、结构完整、证据可追溯的正式报告。适用于项目报告、技术报告、阶段总结、数据分析报告、研究报告和决策简报。

不用于没有事实材料的虚构成果、纯营销文案、法律/医疗/金融最终专业意见或无来源的观点堆砌。

## 2. Trigger Conditions

明确命中：
- 用户要求写项目报告、正式文稿、阶段总结、技术文档、分析报告。
- 用户提供材料并要求组织成摘要、正文、结论和行动建议。
- 用户要求面向特定读者输出报告。

可能命中：
- 仓库分析、数据分析、RAG 设计、法律/医疗/金融信息整理需要正式报告输出。

不应命中：
- 用户只要研究来源综合，使用 `research-synthesis`。
- 用户只要文档摘要，使用 `documentation-analysis`。
- 用户要 PRD，使用 `product-requirements`。

相似分支区别：
- `documentation-analysis` 提取文档信息，本分支组织正式报告。
- `research-synthesis` 比较多来源观点，本分支面向交付文稿。
- 高风险 domain-specific 分支应作为主分支，报告写作作为辅助。

## 3. Required Inputs

必需输入：
- `{{report_goal}}` 报告目的。
- `{{audience}}` 读者。
- `{{materials}}` 事实材料。
- `{{report_type}}` 报告类型。
- `{{length}}` 字数或篇幅。
- `{{tone}}` 语气。

可选输入：
- `{{citation_style}}` 引用格式。
- `{{decision_context}}` 决策背景。
- `{{action_requirements}}` 行动建议要求。
- `{{appendix_materials}}` 附录材料。

缺失时处理：
- 材料不足：先输出提纲和材料缺口，不编造成果。
- 读者未知：标注 `[待补充: audience]`，可默认内部项目干系人。
- 引用要求未知：默认用材料标题、文件名或段落标注。
- 高风险领域：必须保留专业边界和人工复核路径。

## 4. Prompt Construction Rules

- 目标定义：报告必须服务读者决策、记录、复盘或执行。
- 材料整理：先建立材料清单，区分事实、推断、建议和不确定性。
- 结构设计：标题、摘要、背景、方法/依据、主体、结论、行动建议、风险、附录。
- 证据绑定：每个关键结论必须能回溯到材料。
- 行动建议：建议应具体、可执行、分优先级，说明责任人或下一步。
- 引用：按用户要求或默认格式标注来源。
- 边界：不得编造成果、夸大进展或隐藏限制。

## 5. Hard Constraints

- 不得编造项目成果、数据、引用或决策。
- 不得把事实、推断和建议混写成确定结论。
- 不得脱离读者需求堆材料。
- 不得省略材料缺口和不确定性。
- 高风险报告不得给越界专业结论。

## 6. Output Format

最终 prompt 应要求目标模型输出：

```text
标题：
执行摘要：
背景和目的：
材料来源：
主体结构：
关键事实和证据：
分析和结论：
风险与限制：
行动建议：
需补充材料：
附录：
```

## 7. Quality Checklist

- [ ] 是否明确读者和报告目的？
- [ ] 是否要求材料清单和引用来源？
- [ ] 是否区分事实、推断、建议和不确定性？
- [ ] 每个关键结论是否要求绑定证据？
- [ ] 是否包含摘要、正文、结论、风险和行动建议？
- [ ] 是否要求材料不足时输出缺口？
- [ ] 是否控制篇幅、语气和结构？
- [ ] 高风险内容是否保留专业边界？

## 8. Common Mistakes

| Common Mistake | Risk | Repair |
| --- | --- | --- |
| 写成流水账 | 读者无法决策 | 按目的和主题组织结构 |
| 编造成果 | 失真且有责任风险 | 只基于材料写，缺口单列 |
| 没有证据 | 结论不可复核 | 关键结论绑定材料 |
| 建议空泛 | 无法执行 | 写优先级、行动、负责人或下一步 |

## 9. Reusable Template

```text
请基于以下材料写一份报告。

报告目的：{{report_goal}}
读者：{{audience}}
报告类型：{{report_type}}
材料：{{materials}}
篇幅：{{length}}
语气：{{tone}}
引用格式：{{citation_style}}

执行要求：
1. 先建立材料来源清单，标注每份材料的用途。
2. 设计报告结构：标题、摘要、背景、依据、主体、结论、风险、行动建议、附录。
3. 每个关键结论必须绑定材料证据；事实、推断、建议和不确定性分开。
4. 材料不足时输出需补充材料，不编造成果。
5. 行动建议必须具体、可执行，并按优先级排列。

输出格式：
标题、执行摘要、背景和目的、材料来源、主体、关键证据、结论、风险与限制、行动建议、需补充材料、附录。
```

## 10. Example

用户原始需求：

```text
把项目进度材料整理成给负责人看的阶段报告。
```

路由判断：
- 主分支：`documents-research/report-writing`
- 辅助分支：`documents-research/documentation-analysis`
- 风险等级：Low

高质量 prompt：

```text
请基于 `[materials]` 写一份面向项目负责人的阶段报告。目标是说明已完成工作、当前进度、问题风险和下阶段计划。先建立材料来源清单；每个关键结论必须能对应材料，禁止编造未提供成果。输出标题、执行摘要、进度表、关键成果、证据、风险与阻塞、下阶段计划、行动建议和需补充材料，控制在 1500 字以内，语气正式。事实、推断和建议必须分开。
```

质量检查结果：
- [x] 有读者和目的。
- [x] 有证据和材料缺口规则。
- [x] 有摘要、正文、风险和行动建议。
- [x] 禁止编造成果。
