# Report Writing

## 1. Purpose
用于把事实材料组织成面向特定读者的正式报告结构。 本分支只处理该场景专属 prompt 构造；跨场景原则使用 `../../common-principles.md`。

## 2. Trigger Conditions
- 用户要求写项目报告或正式文稿
- 用户要求实习/课程/阶段总结
- 用户要求技术文档

## 3. Required Inputs
- {{report_goal}} 报告目的
- {{audience}} 读者
- {{materials}} 事实材料
- {{tone}} 语气
- {{length}} 字数

缺失信息处理：
- 材料不足时先输出提纲和材料缺口
- 读者未知时按内部项目干系人
- 涉及安全、金钱、医疗、法律、生产数据或大范围改动时，缺失的关键边界必须标注为阻塞问题。
- 非阻塞缺失项使用 `[待补充: field]`，并在最终 prompt 中要求目标模型标注假设。

## 4. Prompt Construction Rules
- 明确读者和目的
- 设计章节结构
- 每个结论绑定事实材料
- 控制字数和语气
- 输出正式文档结构
- 需要适配 Codex、Codex CLI、Claude Code、Gemini CLI 或 ChatGPT 时，必须加入目标工具的工作方式、权限边界和最终报告格式。

## 5. Hard Constraints
- 禁止编造项目成果
- 禁止把建议和事实混写
- 禁止脱离读者需求堆材料
- 必须包含可检查的验收标准。
- 不确定时必须标注假设或提出阻塞问题。

## 6. Output Format
最终 prompt 应要求目标模型输出：
- 标题
- 摘要
- 正文结构
- 事实依据
- 风险/建议
- 附录或材料缺口

## 7. Quality Checklist
- [ ] 读者和目的明确
- [ ] 章节逻辑完整
- [ ] 事实材料可追溯
- [ ] 语气和字数符合要求
- [ ] prompt 包含目标、输入、步骤、约束、输出格式、验收标准和自检。
- [ ] 目标工具的执行环境和限制已写明。

## 8. Common Mistakes
| Common Mistake | Risk | Repair |
| --- | --- | --- |
| 像流水账 | 报告不可读 | 按目标和主题组织 |
| 成果夸大 | 失真 | 绑定材料 |
| 没有结论 | 无法决策 | 每章写结论和下一步 |

## 9. Reusable Template
```text
你是 {{target_ai_tool}}，请处理以下 Report Writing 任务。

任务目标：
{{task_goal}}

输入材料：
- {{report_goal}}
- {{audience}}
- {{materials}}
- {{tone}}
- {{length}}
- {{input_materials}}

执行要求：
1. 明确读者和目的
2. 设计章节结构
3. 每个结论绑定事实材料
4. 控制字数和语气
5. 输出正式文档结构

硬性约束：
- 禁止编造项目成果
- 禁止把建议和事实混写
- 禁止脱离读者需求堆材料
- 必须按 {{output_format}} 输出。
- 必须满足 {{acceptance_criteria}}。

验证与自检：
- 完成后检查目标、输入、约束、输出格式和验收标准是否全部满足。
- 对缺失或不确定的信息标注“假设”或“待补充”。
- 如果无法完成，说明阻塞原因和下一步需要的输入。
```

## 10. Example
用户原始需求：

```text
帮我生成项目阶段报告 prompt。
```

高质量 prompt：

```text
请基于 `[materials]` 写一份面向项目负责人看的阶段报告。目标是说明已完成工作、当前进度、问题风险和下阶段计划。禁止编造未提供成果；每个结论必须能对应材料。输出摘要、进度表、关键成果、风险与阻塞、下阶段计划、需确认事项，控制在 1500 字以内，语气正式。
```
