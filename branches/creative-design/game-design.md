# Game Design

## 1. Purpose
用于为游戏概念、机制、关卡、角色、数值系统、任务系统、剧情、玩家体验、原型开发 prompt 和 GDD 生成强执行 prompt。本分支强调核心玩法循环、可测试原型和 playtest 验收。

## 2. Trigger Conditions
- 用户提到游戏设计、核心玩法、玩法循环、关卡、机制、角色、数值、GDD。
- 用户要求设计游戏概念、任务系统、经济系统、剧情或玩家体验。
- 用户要求输出可开发原型 prompt 或让 Codex 实现游戏原型。

## 3. Required Inputs
- {{game_genre}} 游戏类型。
- {{target_players}} 目标玩家。
- {{platform}} 平台。
- {{core_loop}} 核心玩法循环。
- {{main_mechanics}} 主要机制。
- {{progression_system}} 成长或进度系统。
- {{level_design_goal}} 关卡设计目标。
- {{art_style}} 美术风格。
- {{narrative_scope}} 叙事范围。
- {{prototype_scope}} 原型范围。
- {{success_metrics}} 成功指标。

缺失信息处理：
- 核心循环未知时必须先提出 2-3 个可选核心循环，不直接写世界观。
- 若要求开发实现，必须组合 `coding-feature-development` 或 `cli-agent`，并把原型范围压到可验证最小版本。
- 美术或剧情不足时标注假设，优先保证玩法可测试。

## 4. Prompt Construction Rules
- 必须先定义核心玩法循环：玩家目标、操作、反馈、奖励、下一轮动机。
- 必须说明玩家目标、胜利条件、失败条件、资源和反馈系统。
- 必须区分创意概念、机制设计、内容设计和实现需求。
- 必须要求输出可测试原型范围，限定平台、输入、最小关卡和功能边界。
- 必须说明难度曲线、关卡结构、进度系统和数值假设。
- 必须提供 playtest 验收标准、观察指标和迭代问题。
- 若输出 GDD，必须包含系统、UI/UX、关卡、角色/世界观、数值和原型计划。

## 5. Hard Constraints
- 禁止只写故事不写玩法。
- 禁止机制过多导致无法原型验证。
- 禁止没有目标玩家、平台和核心循环。
- 禁止缺少胜利/失败条件、反馈系统和测试标准。
- 禁止把概念设计和开发实现混在一起不设边界。

## 6. Output Format
最终 prompt 应要求目标模型输出：
- 游戏概念和目标玩家。
- 核心玩法循环。
- 主要机制和反馈。
- 胜利/失败条件。
- 关卡结构和难度曲线。
- 角色/世界观范围。
- 数值和经济系统。
- UI/UX 说明。
- 原型范围。
- playtest 标准和 GDD。

## 7. Quality Checklist
- [ ] 游戏类型、平台和目标玩家明确。
- [ ] 核心玩法循环包含目标、操作、反馈、奖励。
- [ ] 机制数量适合原型验证。
- [ ] 胜利条件和失败条件明确。
- [ ] 难度曲线和关卡目标明确。
- [ ] 叙事和美术不替代玩法设计。
- [ ] 原型范围可开发、可测试。
- [ ] playtest 指标和迭代问题明确。

## 8. Common Mistakes
| Common Mistake | Risk | Repair |
| --- | --- | --- |
| 只有故事没有玩法 | 无法开发或测试 | 先写核心循环和玩家操作 |
| 机制过多 | 原型失焦 | 限定 1-3 个核心机制 |
| 没有目标玩家 | 难度和风格不匹配 | 定义玩家画像和平台 |
| 没有核心循环 | 体验松散 | 写目标、操作、反馈、奖励 |
| 没有反馈系统 | 玩家不知道结果 | 定义视觉、声音、数值和奖励反馈 |
| 没有测试标准 | 不能迭代 | 设计 playtest 指标和成功标准 |
| 概念和开发混在一起 | 执行混乱 | 分 GDD 与原型实现 prompt |

## 9. Reusable Template
```text
你是 {{target_ai_tool}}，请设计游戏。

游戏类型：{{game_genre}}
目标玩家：{{target_players}}
平台：{{platform}}
核心循环：{{core_loop}}
主要机制：{{main_mechanics}}
成长系统：{{progression_system}}
关卡目标：{{level_design_goal}}
美术风格：{{art_style}}
叙事范围：{{narrative_scope}}
原型范围：{{prototype_scope}}
成功指标：{{success_metrics}}

执行要求：
1. 先定义核心玩法循环：目标、操作、反馈、奖励、下一轮动机。
2. 区分概念、机制、内容和实现需求。
3. 设计胜利/失败条件、难度曲线、关卡结构、反馈和数值假设。
4. 输出可测试原型范围，明确平台、输入、最小关卡和非目标。
5. 提供 playtest 验收标准、观察指标和迭代问题。

输出格式：
GDD 摘要、核心循环、机制、关卡、难度、角色/世界观、数值、UI/UX、原型范围、playtest 标准。
```

## 10. Example
用户原始需求：

```text
帮我写一个游戏设计 prompt，输出 GDD 和可测试原型范围。
```

高质量 prompt：

```text
请设计一个 PC 平台 2D roguelike 小游戏，并输出 GDD 和可测试原型范围。目标玩家是喜欢 10-15 分钟短局和构筑选择的玩家。先定义核心循环：探索房间、战斗、获得随机能力、选择升级、进入下一层。必须写清玩家操作、敌人反馈、胜利条件、失败条件、房间结构、难度曲线、3 个核心道具、基础数值假设和 UI/UX。原型范围限制为 1 个角色、3 种敌人、6 个房间、1 个 boss、键鼠控制。输出 playtest 标准：完成率、平均局时、受伤来源、升级选择分布和玩家困惑点。不要只写故事。
```
