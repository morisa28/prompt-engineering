# Security Threat Modeling

## 1. Purpose

用于生成防御性安全 prompt，帮助识别资产、入口、攻击面、信任边界、权限模型、数据泄露风险和加固建议。

不用于攻击性利用链、payload、绕过权限、隐蔽攻击、恶意持久化、规避检测或未授权测试。

## 2. Trigger Conditions

明确命中：
- 用户要求安全审查、威胁建模、权限审计、认证授权检查。
- 用户关注数据泄露、越权访问、输入校验、secrets、日志和审计。
- 用户要求防御性加固、风险清单或安全测试计划。

可能命中：
- RAG、API、数据库、CI/CD、招聘或隐私任务涉及权限和敏感数据。
- 代码审查需要安全维度。

不应命中：
- 用户要求可操作攻击步骤、绕过、提权、持久化、隐蔽或规避检测。
- 用户无法证明授权时，不生成测试执行 prompt，只能给通用防御建议。

相似分支区别：
- `code-review` 审代码质量，本分支聚焦安全风险和防御边界。
- `api-design` 定义接口，本分支审认证、授权和滥用风险。
- `knowledge-base-rag` 处理知识库权限，本分支可辅助 threat model。

## 3. Required Inputs

必需输入：
- `{{system_scope}}` 系统范围。
- `{{assets}}` 资产：数据、账号、凭证、资金、文档等。
- `{{actors}}` 用户角色和外部系统。
- `{{trust_boundaries}}` 信任边界。
- `{{entry_points}}` 入口和接口。
- `{{code_or_docs}}` 代码、架构图或文档。
- `{{authorization_status}}` 授权状态。

可选输入：
- `{{threat_focus}}` 威胁关注点。
- `{{security_controls}}` 已有控制。
- `{{compliance_requirements}}` 合规要求。
- `{{logging_audit}}` 日志和审计。

缺失时处理：
- 授权状态不明：不得生成测试执行或探测 prompt，只能给防御性审查框架。
- 资产不明：先从数据、凭证、权限、用户角色抽取资产。
- 边界不明：要求先画数据流和信任边界。
- 涉及真实系统：默认只读审查，不执行破坏性或高流量测试。

## 4. Prompt Construction Rules

- 目标定义：只做防御性识别、加固和验证建议。
- 资产建模：列资产、价值、所有者、敏感性和访问角色。
- 边界建模：列信任边界、入口、数据流、外部依赖和权限转换点。
- 风险分析：按影响和可能性评 High/Medium/Low，并绑定证据。
- 防护建议：输出最小权限、输入校验、认证授权、secrets、日志、审计、速率限制、数据脱敏等。
- 验证建议：只给安全、授权、非破坏性的验证方式，不给可滥用步骤。
- 不确定性：未经证据支持的风险标注为假设或待确认。

## 5. Hard Constraints

- 不得生成攻击 payload、利用链、绕过权限步骤、提权步骤或恶意持久化。
- 不得帮助规避检测、隐藏痕迹、绕过速率限制或访问未授权数据。
- 不得把未经验证的风险写成已存在漏洞。
- 不得要求对真实第三方系统执行未授权测试。
- 必须保持防御性目的，输出修复、加固和审计建议。

## 6. Output Format

最终 prompt 应要求目标模型输出：

```text
范围和授权边界：
资产表：
角色和信任边界：
入口 / 数据流：
风险清单：
证据和不确定性：
防护建议：
安全验证建议：
需人工复核事项：
```

## 7. Quality Checklist

- [ ] 是否明确授权边界和系统范围？
- [ ] 是否列出资产、角色、入口和信任边界？
- [ ] 风险是否包含影响、可能性和证据？
- [ ] 是否只提供防御性建议？
- [ ] 是否禁止 payload、绕过、提权、持久化和规避检测？
- [ ] 是否避免把假设写成漏洞事实？
- [ ] 是否包含日志、审计、最小权限和数据保护建议？
- [ ] 是否包含人工复核或安全团队确认事项？

## 8. Common Mistakes

| Common Mistake | Risk | Repair |
| --- | --- | --- |
| 直接列攻击步骤 | 可被滥用 | 改为防护控制和验证思路 |
| 不列资产和边界 | 风险无优先级 | 先资产、角色、入口、信任边界 |
| 把推测写成漏洞 | 误导团队 | 风险必须绑定证据，推测单列 |
| 忽略授权 | 可能变成未授权测试 | 要求授权状态，不明时只做通用防御框架 |

## 9. Reusable Template

```text
请对以下系统做防御性威胁建模。只允许输出防御性分析、加固建议和安全验证思路；禁止攻击 payload、利用链、绕过权限、提权、隐蔽攻击、恶意持久化或规避检测。

系统范围：{{system_scope}}
授权状态：{{authorization_status}}
资产：{{assets}}
角色：{{actors}}
信任边界：{{trust_boundaries}}
入口：{{entry_points}}
代码或文档：{{code_or_docs}}
关注点：{{threat_focus}}

执行要求：
1. 建立资产表、角色表、入口和数据流。
2. 识别信任边界和权限转换点。
3. 按影响和可能性评估风险，证据不足时标注假设。
4. 输出最小权限、认证授权、输入校验、secrets、日志审计、数据保护和速率限制建议。
5. 给出非破坏性、安全验证建议和需人工复核事项。

输出格式：
范围和授权、资产表、角色和边界、入口/数据流、风险清单、证据和不确定性、防护建议、验证建议、人工复核。
```

## 10. Example

用户原始需求：

```text
写一个 prompt，让 AI 给我的 API 做威胁建模，只要防御建议。
```

路由判断：
- 主分支：`software-engineering/security-threat-modeling`
- 辅助分支：`software-engineering/code-review`, `software-engineering/cli-agent`
- 风险等级：High

高质量 prompt：

```text
请对 `/repo/api` 做防御性威胁建模。授权状态：项目内部授权审查。先读取认证、授权、输入校验、日志、secrets 和数据访问相关文件；识别资产、入口、信任边界、角色权限和数据流。按 High/Medium/Low 输出风险、影响、可能性、证据、修复建议和非破坏性验证方式。禁止生成攻击 payload、绕过步骤、提权方法、隐蔽攻击、恶意持久化或规避检测说明；未经证据支持的内容必须标注为假设。
```

质量检查结果：
- [x] 防御性边界明确。
- [x] 包含资产、入口和边界。
- [x] 禁止攻击性细节。
- [x] 风险要求证据。
