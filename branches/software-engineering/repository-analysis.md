# Repository Analysis

## 1. Purpose

用于让 agent 在只读模式下建立仓库理解，输出结构、技术栈、入口、核心模块、数据流、依赖关系、运行方式、测试方式、风险点和改进建议。

不用于直接修 bug、实现功能、改代码或做未授权重构。

## 2. Trigger Conditions

明确命中：
- 用户要求“分析整个 repo”“理解项目结构”“找入口文件”“生成项目说明”。
- 用户要求评估仓库风险、技术债、模块关系、运行方式。
- 用户明确说不要修改文件。

可能命中：
- 用户要 onboarding 文档、架构概览或交接报告。
- 用户要基于仓库生成后续开发建议。

不应命中：
- 用户要修复具体错误，使用 `bugfix-debugging`。
- 用户要实现新功能，使用 `coding-feature-development`。
- 用户要 review PR/diff，使用 `code-review`。

相似分支区别：
- `documentation-analysis` 分析文档，不一定分析代码结构。
- `report-writing` 组织报告，可作为辅助输出格式。
- `plan-mode` 为未来修改做计划，但本分支核心是只读理解。

## 3. Required Inputs

必需输入：
- `{{working_directory}}` 仓库路径。
- `{{analysis_goal}}` 分析目标。
- `{{focus_areas}}` 关注模块或问题。
- `{{output_depth}}` 概览、详细或交接级。

可选输入：
- `{{excluded_paths}}` 不读路径。
- `{{known_questions}}` 用户已有疑问。
- `{{report_audience}}` 报告读者。
- `{{time_budget}}` 大仓库抽样策略。

缺失时处理：
- 未指定关注点：默认覆盖结构、技术栈、入口、运行、测试、核心模块、风险。
- 大型仓库：先读 README、配置、顶层目录和明显入口，再说明抽样范围。
- 缺少工作目录：标记 `[待补充: working_directory]`。
- 用户要求分析私密或敏感仓库：要求不要输出 secrets、token、个人数据或内部敏感片段。

## 4. Prompt Construction Rules

- 目标定义：明确输出仓库理解报告，不修改文件。
- 上下文整理：先读 README、依赖配置、构建/测试配置、入口文件、目录树。
- 任务分解：建立文件树，识别技术栈，定位入口，分析模块职责和数据流，提取命令，列风险和建议。
- 证据规则：技术栈、入口、命令和模块判断必须绑定文件证据。
- 输出格式约束：用项目概览、文件树、模块表、数据流、依赖、命令、风险、建议、不确定性分节。
- 验收标准：报告能让新成员知道如何运行、测试、定位核心模块和下一步改进。
- 工具适配：Codex/CLI agent 必须写明只读、禁止修改文件、禁止运行破坏性命令。

## 5. Hard Constraints

- 禁止修改文件。
- 禁止运行破坏性命令或格式化命令。
- 禁止把未读取的目录写成确定结论。
- 禁止输出 secrets、密钥、个人数据或敏感配置值。
- 禁止生成与项目无关的通用教程。
- 不确定时必须标注“推断”或“未确认”。

## 6. Output Format

最终 prompt 应要求目标模型输出：

```text
项目概览：
读取范围：
目录树：
技术栈和证据：
入口文件和运行命令：
核心模块表：
数据流 / 控制流：
外部依赖：
测试与构建：
风险点：
改进建议：
不确定性和未读取区域：
```

## 7. Quality Checklist

- [ ] 是否明确只读不改？
- [ ] 是否要求读取 README、配置、入口和测试配置？
- [ ] 是否要求输出目录树和核心模块表？
- [ ] 技术栈、命令和入口是否要求文件证据？
- [ ] 是否区分事实、推断和未读取区域？
- [ ] 是否包含数据流或控制流说明？
- [ ] 是否输出风险点和后续建议但不执行修改？
- [ ] 是否避免泄露 secrets 或敏感信息？

## 8. Common Mistakes

| Common Mistake | Risk | Repair |
| --- | --- | --- |
| 只列文件树不解释职责 | 用户仍无法理解项目 | 输出核心模块表和数据流 |
| 不读配置文件就判断技术栈 | 技术栈误判 | 要求读取 package、pyproject、Cargo、pom 等配置 |
| 把猜测写成事实 | 误导后续开发 | 事实、推断、未读取区域分开 |
| 顺手改代码 | 违背只读分析 | 明确禁止修改和破坏性命令 |

## 9. Reusable Template

```text
你是 {{target_ai_tool}}，请只读分析 {{working_directory}}，不要修改任何文件。

分析目标：{{analysis_goal}}
关注重点：{{focus_areas}}
输出深度：{{output_depth}}
排除路径：{{excluded_paths}}
读者：{{report_audience}}

执行步骤：
1. 读取 README、依赖配置、构建/测试配置、入口文件和顶层目录。
2. 建立简化文件树，标注核心路径和未读取区域。
3. 识别技术栈、运行命令、测试命令、核心模块、数据流和外部依赖。
4. 技术栈、命令、入口和模块判断必须引用文件证据。
5. 输出风险点、改进建议和后续可执行任务，但不要执行修改。

硬性约束：
- 禁止修改文件或运行破坏性命令。
- 禁止输出 secrets、token、个人数据或敏感配置值。
- 未读取路径不得写成确定结论。

输出格式：
项目概览、读取范围、目录树、技术栈证据、入口与命令、核心模块表、数据流、测试构建、风险、建议、不确定性。
```

## 10. Example

用户原始需求：

```text
分析这个 GitHub 仓库，告诉我入口、技术栈、模块关系和改进建议，不要改代码。
```

路由判断：
- 主分支：`software-engineering/repository-analysis`
- 辅助分支：`documents-research/report-writing`
- 风险等级：Medium

高质量 prompt：

```text
你是 Codex，请只读分析 `/repo/app`，不要修改文件。先读取 README、`package.json`、构建/测试配置、入口文件和顶层目录；建立目录树并识别技术栈、入口、核心模块、数据流、运行命令和测试方式。技术栈和命令必须绑定文件证据。输出项目概览、读取范围、目录结构、关键模块表、数据流、风险点、改进建议和未读取区域。禁止运行破坏性命令、输出 secrets 或把未读目录写成确定结论。
```

质量检查结果：
- [x] 明确只读。
- [x] 要求文件证据。
- [x] 包含模块、数据流、风险和建议。
- [x] 禁止泄露敏感信息。
