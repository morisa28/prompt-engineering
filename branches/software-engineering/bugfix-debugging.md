# Bugfix Debugging

## 1. Purpose

用于让代码 agent 根据日志、复现步骤、环境和相关源码定位根因，并用最小安全改动修复错误。适用于构建失败、测试失败、运行时异常、页面白屏、依赖冲突、CLI 报错和回归缺陷。

不用于从零开发新功能、大范围架构重写、纯仓库理解或没有具体失败信号的泛化“优化项目”。

## 2. Trigger Conditions

明确命中：
- 用户提供错误日志、堆栈、失败命令或失败测试。
- `npm run build`、`npm test`、`pytest`、`cargo test`、CI job 等失败。
- 页面白屏、接口 500、CLI 命令失败、依赖安装失败。

可能命中：
- 用户说“最近改完坏了”“帮我修这个 bug”，但日志不完整。
- 用户要求修复 CI 构建失败，主分支可为 `devops-ci`，本分支作为辅助。

不应命中：
- 用户要新增功能，使用 `coding-feature-development`。
- 用户只想理解仓库，使用 `repository-analysis`。
- 用户只要测试策略，使用 `test-generation`。

相似分支区别：
- `code-review` 找风险但不一定修改。
- `refactor-architecture` 改结构但不以失败日志为中心。
- `devops-ci` 以 pipeline、runner、部署环境为中心。

## 3. Required Inputs

必需输入：
- `{{working_directory}}` 工作目录。
- `{{error_log}}` 完整日志、堆栈或失败断言。
- `{{reproduction_steps}}` 复现步骤或失败命令。
- `{{environment}}` OS、语言版本、package manager、框架版本、CI runner 等。
- `{{expected_behavior}}` 期望行为。
- `{{actual_behavior}}` 实际行为。

可选输入：
- `{{recent_changes}}` 最近改动、PR、diff 或相关提交。
- `{{related_files}}` 用户怀疑的文件。
- `{{verification_commands}}` 构建、测试、lint 或手动验证命令。

缺失时处理：
- 日志缺失或不完整：标记为阻塞或要求补充关键栈；可先生成带 `[待补充: error_log]` 的 prompt。
- 环境未知：可要求 agent 从配置识别，并把版本假设写入报告。
- 复现步骤未知：必须让 agent 先尝试用失败命令复现，不能直接猜修复。
- 涉及生产、数据丢失、安全、支付或权限问题：缺失影响范围和回滚方案时阻塞。

## 4. Prompt Construction Rules

- 目标定义：明确修复哪个失败路径，而不是“优化项目”。
- 上下文整理：要求先读配置、失败日志、栈中源码、相关测试和最近改动。
- 根因分析：要求区分代码、配置、依赖、环境、数据、缓存和测试本身问题。
- 任务分解：复现失败，定位根因，提出最小方案，实施修复，运行验证，报告结果。
- 约束注入：禁止无关重构、全仓库格式化、删除测试、降低断言、注释掉失败代码、隐藏错误。
- 输出格式约束：必须输出根因证据、修改文件、关键改动、验证命令和剩余风险。
- 验收标准：原失败命令通过，相关测试通过，改动范围最小，无未说明副作用。
- 自检步骤：检查是否使用日志和复现步骤、是否有根因证据、是否验证原失败路径。
- 工具适配：Codex/CLI agent 必须包含工作目录、可运行命令、权限边界和最终报告格式。

## 5. Hard Constraints

- 不得编造日志、文件、命令结果或测试结果。
- 不得删除、跳过或弱化测试来掩盖失败。
- 不得通过注释业务逻辑、吞掉异常或扩大 try/catch 掩盖根因。
- 不得做无关重构、样式调整、依赖升级或全仓库格式化。
- 不得回退用户改动，除非用户明确要求。
- 需要新增依赖、删除文件、改数据库或改生产配置时必须先说明风险并等待确认。

## 6. Output Format

最终 prompt 应要求目标模型输出：

```text
根因：
证据：
修改文件：
关键改动：
验证命令与结果：
未验证项：
剩余风险：
```

## 7. Quality Checklist

- [ ] 是否包含完整日志或明确 `[待补充: error_log]`？
- [ ] 是否包含复现步骤或失败命令？
- [ ] 是否要求先复现或读取证据再修改？
- [ ] 是否要求根因绑定日志、文件或命令证据？
- [ ] 是否明确最小修复和禁止无关重构？
- [ ] 是否禁止删除测试、降低断言和掩盖错误？
- [ ] 是否包含验证命令并覆盖原失败路径？
- [ ] 是否要求最终变更摘要和剩余风险？

## 8. Common Mistakes

| Common Mistake | Risk | Repair |
| --- | --- | --- |
| 看到报错就猜测修改位置 | 修错文件或引入新 bug | 要求先复现、读栈和相关源码，根因必须有证据 |
| 把 bugfix 写成重构任务 | 改动过大，难以验证 | 明确只修复当前失败路径，其他问题列为后续建议 |
| 允许删除测试或降低断言 | 掩盖真实缺陷 | 明确禁止，并要求新增或保留回归测试 |
| 不要求验证命令 | 无法确认修复有效 | 写明原失败命令、相关测试和无法运行时的报告方式 |

## 9. Reusable Template

```text
你是 {{target_ai_tool}}，请在 {{working_directory}} 修复以下错误。

错误日志：
{{error_log}}

复现步骤：
{{reproduction_steps}}

环境：
{{environment}}

期望行为：
{{expected_behavior}}

实际行为：
{{actual_behavior}}

最近改动/相关文件：
{{recent_changes}}
{{related_files}}

执行要求：
1. 先读取项目配置、失败日志、堆栈指向文件、相关测试和最近改动。
2. 先复现或解释无法复现的原因，再定位根因。
3. 区分代码、配置、依赖、环境、数据或测试问题。
4. 只做最小修复；禁止无关重构、全仓库格式化、新增依赖、删除测试、降低断言或注释掉失败逻辑。
5. 修复后运行 {{verification_commands}}，至少覆盖原失败路径。

输出格式：
- 根因：
- 证据：
- 修改文件：
- 关键改动：
- 验证命令与结果：
- 未验证项：
- 剩余风险：

自检：
- 日志和复现步骤是否被使用？
- 修复是否最小？
- 是否验证原失败路径？
- 是否保留或新增回归保护？
```

## 10. Example

用户原始需求：

```text
帮我让 Codex 修复 npm run build 报错，日志里说 Cannot find module './config'，修完要跑测试，不要乱改。
```

路由判断：
- 主分支：`software-engineering/bugfix-debugging`
- 辅助分支：`software-engineering/test-generation`, `software-engineering/cli-agent`
- 风险等级：Medium

高质量 prompt：

```text
你是 Codex，请在 `/repo/app` 修复 `npm run build` 报错。错误日志：`Cannot find module './config'`，复现步骤：运行 `npm run build`。先读取 `package.json`、构建配置、报错栈指向文件和相关测试；复现失败后定位根因，说明是路径、导出、依赖或构建配置问题。只做最小修复，禁止无关重构、全仓库格式化、新增依赖、删除测试或降低断言。修复后运行 `npm run build` 和相关测试。最终输出根因、证据、修改文件、关键改动、验证命令与结果、剩余风险。
```

质量检查结果：
- [x] 包含失败命令和错误日志。
- [x] 要求根因证据和最小修复。
- [x] 明确禁止删除测试和无关重构。
- [x] 包含构建和测试验证。
