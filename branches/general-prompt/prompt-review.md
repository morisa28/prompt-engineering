# Prompt Review

## 1. Purpose
用于评审用户已经写好的 prompt，判断它为什么执行不好，并把问题拆成可修复的结构缺口。 本分支只处理该场景专属 prompt 构造；跨场景原则使用 `../../common-principles.md`。

## 2. Trigger Conditions
- 用户说“看看这个 prompt 哪里不好”
- 用户要求评分、诊断、找歧义点
- 用户已有 prompt 但不确定为什么结果不稳定

## 3. Required Inputs
- {{original_prompt}} 原 prompt 全文
- {{target_ai_tool}} 目标工具
- {{expected_result}} 期望结果
- {{known_failures}} 已出现的问题或坏输出

缺失信息处理：
- 目标工具未知时按 ChatGPT 通用 prompt 评审
- 期望结果未知时只评估结构，不猜业务目标
- 涉及安全、金钱、医疗、法律、生产数据或大范围改动时，缺失的关键边界必须标注为阻塞问题。
- 非阻塞缺失项使用 `[待补充: field]`，并在最终 prompt 中要求目标模型标注假设。

## 4. Prompt Construction Rules
- 先诊断再重写，诊断必须覆盖目标、上下文、输入、约束、输出格式、验收标准、风险控制、自检八项
- 每个问题必须写出风险和修复方式
- 优化版必须保留用户原意，不加入无依据的新任务
- 歧义点必须单独列出，并标明是否阻塞重写
- 需要适配 Codex、Codex CLI、Claude Code、Gemini CLI 或 ChatGPT 时，必须加入目标工具的工作方式、权限边界和最终报告格式。

## 5. Hard Constraints
- 禁止直接给重写版而不解释问题
- 禁止把用户未提供的事实写成确定条件
- 必须保留原 prompt 的核心目标和语气边界
- 必须包含可检查的验收标准。
- 不确定时必须标注假设或提出阻塞问题。

## 6. Output Format
最终 prompt 应要求目标模型输出：
- 评分表
- 问题诊断表
- 歧义点
- 优化后的 prompt
- 改动理由
- 仍需补充的信息

## 7. Quality Checklist
- [ ] 八个评审维度全部覆盖
- [ ] 每个扣分点都有可执行修复方式
- [ ] 优化版能直接复制使用
- [ ] 未把假设写成事实
- [ ] prompt 包含目标、输入、步骤、约束、输出格式、验收标准和自检。
- [ ] 目标工具的执行环境和限制已写明。

## 8. Common Mistakes
| Common Mistake | Risk | Repair |
| --- | --- | --- |
| 只说不够清楚 | 用户不知道该改哪里 | 指出缺失字段和具体补法 |
| 直接重写 | 掩盖原 prompt 的根因 | 先诊断再重写 |
| 加入新需求 | 偏离用户原意 | 新增内容只能作为假设或可选项 |

## 9. Reusable Template
```text
你是 {{target_ai_tool}}，请处理以下 Prompt Review 任务。

任务目标：
{{task_goal}}

输入材料：
- {{original_prompt}}
- {{target_ai_tool}}
- {{expected_result}}
- {{known_failures}}
- {{review_dimensions}}
- {{input_materials}}

执行要求：
1. 先诊断再重写，诊断必须覆盖目标、上下文、输入、约束、输出格式、验收标准、风险控制、自检八项
2. 每个问题必须写出风险和修复方式
3. 优化版必须保留用户原意，不加入无依据的新任务
4. 歧义点必须单独列出，并标明是否阻塞重写

硬性约束：
- 禁止直接给重写版而不解释问题
- 禁止把用户未提供的事实写成确定条件
- 必须保留原 prompt 的核心目标和语气边界
- 必须按 {{output_format}} 输出。
- 必须满足 {{acceptance_criteria}}。

验证与自检：
- 完成后检查目标、输入、约束、输出格式和验收标准是否全部满足。
- 对缺失或不确定的信息标注“假设”或“待补充”。
- 如果无法完成，说明阻塞原因和下一步需要的输入。
```

## 10. Example
用户原始需求：

```text
帮我看看这个 prompt 哪里不好：`帮我把这个项目优化一下。`
```

高质量 prompt：

```text
请评审以下 prompt：`帮我把这个项目优化一下。`
目标工具：Codex。
请先按目标、上下文、输入、约束、输出格式、验收标准、风险控制、自检八个维度评分并解释扣分原因；再列出歧义点和可能导致误改的风险；最后给出一个保留原意的 Codex 版 prompt。重写版必须包含工作目录、需要先读取的文件、允许和禁止修改范围、验证命令、最终报告格式。不要加入用户未要求的新功能。
```
